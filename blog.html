project writeup

I built a GPU from resistors for 700 rupees

Priyanka Tiwari  |  GXU-1: Graphics Execution Unit, Revision 1

This started as a question I could not stop thinking about. Modern IoT devices collect data and report it. They do not compute. They definitely do not render graphics. And if you want a display output on an embedded system, you are basically forced to buy a dedicated display controller IC, an HDMI driver, or a tiny SPI LCD that shows you exactly nothing interesting at exactly nothing interesting resolution.

I wanted to know if you could skip all of that. Could you take a microcontroller, some discrete resistors, and build something that genuinely deserves to be called a GPU? Not a display module. Not a framebuffer wrapper. An actual programmable graphics processor with a defined instruction set, a rasterizer, and network-addressable command bus.

GXU-1 is the answer. It drives a standard VGA monitor at 320x240 at 60Hz, executes a custom 8-opcode binary ISA sent over WiFi, and costs under 700 rupees in components. Here is how it works.

Before anything else: the architecture comparison that motivated the whole thing.

GXU-1 vs RTX 4090: same concept, different scale

The hardware problem

VGA is an analog standard from 1987. Your monitor expects three analog voltage signals (Red, Green, Blue) between 0V and 0.7V, plus two digital sync pulses that tell it when each line ends and when each frame ends. The problem is that a microcontroller only outputs digital signals: 0V or 3.3V. Nothing in between.

The standard solution is a DAC chip. The GXU-1 solution is a resistor ladder.

An R-2R resistor ladder uses only two resistor values arranged in a repeating pattern to convert digital bits into proportional analog voltages. For GXU-1, each color channel gets 2 bits, giving 4 voltage levels per channel, 64 total colors. The whole DAC uses 18 resistors at 270 ohms and 9 at 560 ohms. Total cost: 35 rupees.

R-2R DAC output voltages per 2-bit value

The ESP32-S3 was chosen specifically because it has a hardware LCD parallel interface. This lets the DMA controller drive all 8 GPIO pins simultaneously in sync with VGA timing at 25.175 MHz, without any CPU involvement. The original ESP32 and ESP32-C3 do not have this peripheral. It is the only reason 60Hz is achievable.

The instruction set

The thing that makes GXU-1 a GPU instead of a display controller is the ISA. A display controller takes pixels and outputs them. A GPU takes drawing commands, figures out which pixels correspond to those commands, and writes them to the framebuffer. That process is rasterization, and it requires a defined command language.

GXU-1 has 8 opcodes. Every packet starts with a 1-byte opcode, followed by arguments encoded as 16-bit big-endian unsigned integers. Color is a single byte with 2 bits per channel.

The choice of binary encoding over something like JSON is not aesthetic. It is a performance requirement. Here is the actual comparison for a single line command:

packet size: binary ISA vs JSON encoding (same command)

A JSON-encoded line command is about 58 bytes and takes several hundred microseconds to parse. The binary packet is 10 bytes and parses in under 5 microseconds. At the framerate GXU-1 targets, that difference is the gap between working and not working.

The dual-core architecture

Most microcontroller projects run everything on one core. WiFi, computation, display output, all sharing the same CPU time. This is why IoT devices stutter when they are busy doing anything non-trivial.

GXU-1 splits responsibilities hard across the two cores of the ESP32-S3. Core 0 and Core 1 never touch each other's domain. The only thing connecting them is a FreeRTOS queue that acts as the internal command bus.

[Browser / Python script / any WiFi device] | WebSocket over WiFi | [Core 0 -- Command Processor] WiFi stack + WebSocket server ISA packet decoder (isa_decode) FreeRTOS queue writer Telemetry JSON sender (200ms interval) | xQueueSend(gxu_queue, &cmd) | [Core 1 -- GX Core (Rasterizer)] FreeRTOS queue reader Bresenham line, rect fill, blit algorithms VRAM writes (76800 byte framebuffer) Tile heatmap counters (16x12 grid) | DMA | [8 GPIO pins -- LCD parallel bus] | [R-2R Resistor DAC] 270 ohm + 560 ohm resistor ladder 0V to 0.7V analog per channel | [VGA Monitor] 320x240 at 60Hz analog RGB + H-Sync + V-Sync

The DMA hardware streams VRAM to GPIO pins continuously, completely independent of both cores. This means a burst of incoming WiFi packets on Core 0 never causes display flicker, and a heavy rendering operation on Core 1 never delays network responses. The queue absorbs any burst with 64-command depth.

Total perceived latency is somewhere between 10ms and 25ms. The human eye stops noticing delays below about 100ms for display feedback, so this is comfortably imperceptible.

The VRAM heatmap

Every time Core 1 writes a pixel, it increments a counter for the tile that pixel falls in. The 320x240 screen is divided into 192 tiles (16 columns by 12 rows, each tile is 20x20 pixels). Core 0 reads these counters every 200ms and includes them in the telemetry JSON it sends back to the browser.

The dashboard renders them as a color grid: cold tiles are dark blue, warm tiles shift through green, hot tiles go amber and red. AMD's Radeon GPU Profiler shows the exact same visualization for production GPU workloads. GXU-1 runs the same concept on a 450-rupee chip.

What I actually learned

The resistor ladder is the part that surprised me most. I knew the theory but building it and then measuring the voltage with a multimeter and seeing exactly 0.35V where the math said 0.35V should be is a genuinely satisfying thing. Physics is consistent, it turns out.

The DMA timing took the longest to get right. The VGA standard has very specific timing requirements around the horizontal and vertical porch periods, the sync pulse widths, the pixel clock. Getting the LCD peripheral configured with the correct values took a lot of trial and error. If the display shows white noise, the porch values are wrong. If colors are shifted, the GPIO assignment is wrong. If the image tears, the sync is off. Each symptom points to exactly one thing.

The FreeRTOS queue architecture taught me more about real-time systems than any course would have. The decision to put WiFi and rendering on separate cores with a defined inter-core communication channel is the same decision that CPU-GPU interface designers make, just at a different scale. Core 0 is the CPU, Core 1 is the GPU, the queue is the PCIe bus. The concepts are identical.

And the ISA design itself was the most interesting part. Every decision compounds: binary vs text, big vs little endian, 2-bit vs 4-bit color channels, packet size vs expressiveness. Real GPU ISA designers make the same tradeoffs at much higher stakes. Working through them at this scale makes it clear why the decisions they made are the ones they made.

Try the software emulator

If you do not have the hardware, the complete GXU-1 ISA is implemented in a browser-based emulator using the Canvas API. It includes the VRAM heatmap, live telemetry, and a raw hex input where you can type ISA packets directly. To draw a red diagonal line from corner to corner, type: 

03 00 00 00 F0 01 3F 00 EF 0C

That is opcode 0x03 (GXU_LINE), x0=0, y0=240, x1=319, y1=239, color=0x0C (red). Hit execute and watch it rasterize.
